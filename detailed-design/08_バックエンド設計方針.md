# バックエンド設計方針

> バージョン: 1.0.0
> 作成日: 2026-03-07
> 対象: Synapse バックエンド開発チーム全員

---

## 1. 目的・スコープ

### 目的

バックエンド実装が本格化する前に、**全員が従うべきルール**を明文化する。
統一されていない実装は以下の問題を引き起こすため、事前合意が必要：

- レイヤー間の依存関係が崩れることによる保守コストの増大
- 命名規約の揺れによる可読性低下
- エラーハンドリング・ログの不統一による障害調査の困難化

### このドキュメントで決定すること

- クリーンアーキテクチャのレイヤー構成と依存ルール
- CQRS / MediatR の実装パターン
- ドメインモデルの設計指針（Entity / Value Object / Domain Exception）
- インフラ層の実装方針（EF Core / Dapper / Repository）
- API 層の実装方針（Controller / ミドルウェア / エラーハンドリング）
- 認証・認可（JWT / RBAC）
- リアルタイム通信（SignalR）
- バックグラウンドジョブ（Hangfire）
- ログ（Serilog）
- 命名規約
- コメント規約（日本語コメント）
- テスト方針

### このドキュメントで決定しないこと

- 各エンドポイントの詳細なリクエスト/レスポンス定義（`detailed-design/05_API仕様書.md` を参照）
- データベースのテーブル定義（`basic-design/04_ER図.md` を参照）
- 業務ルール・機能要件（`requirements/` を参照）

## 2. アーキテクチャ概要

### クリーンアーキテクチャ

```
┌────────────────────────────────────────┐
│            Synapse.API                 │  ← コントローラ・DI・エントリポイント
│  ┌──────────────────────────────────┐  │
│  │      Synapse.Application         │  │  ← ユースケース（Commands / Queries）
│  │  ┌────────────────────────────┐  │  │
│  │  │     Synapse.Domain         │  │  │  ← エンティティ・値オブジェクト・ドメインロジック
│  │  └────────────────────────────┘  │  │
│  └──────────────────────────────────┘  │
│  Synapse.Infrastructure                │  ← DB・外部サービス実装（Application を実装）
└────────────────────────────────────────┘
```

### 依存ルール（厳守）

| 依存元 | 依存先 | 可否 |
|--------|--------|------|
| Domain | Application / Infrastructure / API | **禁止** |
| Application | Domain | **可** |
| Application | Infrastructure | **禁止**（インターフェース経由のみ） |
| Infrastructure | Application / Domain | **可** |
| API | Application / Domain / Infrastructure | **可** |

- **Domain は他の全レイヤーに依存しない**。これが最重要ルール
- Application は Infrastructure の具体実装を知らない。インターフェースに依存する
- DI コンテナへの登録は API プロジェクトで行う

## 4. CQRS / MediatR パターン

### 基本方針

- **書き込み（副作用あり）** → `IRequest<T>` を実装した `Command`
- **読み取り（副作用なし）** → `IRequest<T>` を実装した `Query`
- ハンドラーは 1 ユースケース = 1 ハンドラーとする
- コントローラーは `_mediator.Send()` を呼ぶだけ。ビジネスロジックをコントローラーに書かない

### Command の実装パターン

```csharp
// Command 定義（Application/Commands/ManufacturingOrders/CreateManufacturingOrder/）
public record CreateManufacturingOrderCommand(
    string ProductionPlanId,
    string ItemCode,
    decimal Quantity,
    DateTimeOffset PlannedStartDate,
    DateTimeOffset PlannedEndDate
) : IRequest<string>;  // 戻り値は生成した ID

// Handler 実装
public class CreateManufacturingOrderCommandHandler
    : IRequestHandler<CreateManufacturingOrderCommand, string>
{
    private readonly IManufacturingOrderRepository _repository;
    private readonly IUnitOfWork _unitOfWork;

    public CreateManufacturingOrderCommandHandler(
        IManufacturingOrderRepository repository,
        IUnitOfWork unitOfWork)
    {
        _repository = repository;
        _unitOfWork = unitOfWork;
    }

    public async Task<string> Handle(
        CreateManufacturingOrderCommand request,
        CancellationToken cancellationToken)
    {
        var order = ManufacturingOrder.Create(
            request.ItemCode,
            request.Quantity,
            request.PlannedStartDate,
            request.PlannedEndDate);

        await _repository.AddAsync(order, cancellationToken);
        await _unitOfWork.SaveChangesAsync(cancellationToken);

        return order.Id;
    }
}
```

### Query の実装パターン

```csharp
// Query 定義
public record GetManufacturingOrderListQuery(
    string? Status,
    DateTimeOffset? From,
    DateTimeOffset? To,
    int Page,
    int PageSize
) : IRequest<PagedResult<ManufacturingOrderListDto>>;

// Handler 実装（読み取りは Dapper を使用可）
public class GetManufacturingOrderListQueryHandler
    : IRequestHandler<GetManufacturingOrderListQuery, PagedResult<ManufacturingOrderListDto>>
{
    private readonly IDbConnection _db;

    public GetManufacturingOrderListQueryHandler(IDbConnection db)
    {
        _db = db;
    }

    public async Task<PagedResult<ManufacturingOrderListDto>> Handle(
        GetManufacturingOrderListQuery request,
        CancellationToken cancellationToken)
    {
        // Dapper で直接 SQL を発行
        // ...
    }
}
```

### MediatR パイプラインビヘイビア

以下のビヘイビアを登録する（実行順序は登録順）：

| ビヘイビア | 役割 |
|---|---|
| `LoggingBehavior<TRequest, TResponse>` | リクエスト/レスポンスの構造化ログ出力 |
| `ValidationBehavior<TRequest, TResponse>` | FluentValidation によるバリデーション |
| `TransactionBehavior<TRequest, TResponse>` | Command のトランザクション管理（Query には適用しない） |

### DTO 設計規約

#### 基本ルール

- DTO は `record` で定義する（イミュータブル・等値比較・分割代入が可能）
- DTO はドメインエンティティに依存しない。プリミティブ型のみで構成する
- `null` を返す可能性があるフィールドは `T?` で明示する
- DTO の命名: `{対象}{用途}Dto`（例: `ManufacturingOrderListDto`、`ManufacturingOrderDetailDto`）

#### C# / JSON / TypeScript 型対応表

フロントエンドの型定義はこの対応表に従う。型の不一致は実行時エラーの温床になるため厳守する。

| C# 型 | JSON 値の例 | TypeScript 型 |
|---|---|---|
| `string`（ULID） | `"01ARZ3NDEKTSV4RRFFQ69G5FAV"` | `Ulid`（`string` エイリアス） |
| `string` | `"テキスト"` | `string` |
| `int` / `long` | `123` | `number` |
| `decimal` | `123.45` | `number` |
| `bool` | `true` / `false` | `boolean` |
| `DateTimeOffset` | `"2026-03-07T10:00:00Z"` | `IsoDateTimeString`（`string` エイリアス） |
| `DateOnly` | `"2026-03-07"` | `IsoDateString`（`string` エイリアス） |
| `enum`（camelCase） | `"issued"` | `'draft' \| 'issued' \| ...`（union type） |
| `T?`（nullable） | `null` または値 | `T \| null` |
| `IEnumerable<T>` | `[...]` | `T[]` |

#### DTO 実装例

```csharp
// 一覧用 DTO（必要最小限のフィールドのみ）
public record ManufacturingOrderListDto(
    string Id,                           // ULID → TypeScript: Ulid
    string ItemCode,
    string ItemName,
    decimal Quantity,
    DateOnly PlannedDate,                // → TypeScript: IsoDateString
    DateTimeOffset PlannedStartDate,     // → TypeScript: IsoDateTimeString
    ManufacturingOrderStatus Status,     // → TypeScript: ManufacturingOrderStatus (union type)
    DateTimeOffset CreatedAt             // → TypeScript: IsoDateTimeString
);

// 詳細用 DTO（一覧用より多くのフィールドを持つ）
public record ManufacturingOrderDetailDto(
    string Id,
    string ItemCode,
    string ItemName,
    decimal Quantity,
    DateOnly PlannedDate,
    DateTimeOffset PlannedStartDate,
    DateTimeOffset PlannedEndDate,
    ManufacturingOrderStatus Status,
    string? WorkInstruction,             // nullable → TypeScript: string | null
    DateTimeOffset CreatedAt,
    DateTimeOffset UpdatedAt
);
```

## 5. ドメイン設計

### エンティティ

- 集約ルートは `AggregateRoot` 基底クラスを継承する
- ID は `string`（ULID を使用）とする
- `public` セッターは原則禁止。状態変更はドメインメソッド経由で行う
- ドメインイベントは集約ルートに蓄積し、コミット前に発行する（将来対応）

```csharp
public class ManufacturingOrder : AggregateRoot
{
    public string ItemCode { get; private set; }
    public decimal Quantity { get; private set; }
    public ManufacturingOrderStatus Status { get; private set; }

    // ファクトリメソッドで生成する（コンストラクタを private/protected に）
    public static ManufacturingOrder Create(
        string itemCode, decimal quantity, DateTime plannedStart, DateTime plannedEnd)
    {
        // ドメインバリデーション
        if (quantity <= 0)
            throw new DomainException("数量は1以上でなければなりません。");

        return new ManufacturingOrder
        {
            Id = Ulid.NewUlid().ToString(),
            ItemCode = itemCode,
            Quantity = quantity,
            Status = ManufacturingOrderStatus.Draft,
        };
    }

    // 状態変更はドメインメソッドで表現する
    public void Issue()
    {
        if (Status != ManufacturingOrderStatus.Draft)
            throw new DomainException("下書き状態の製造指示のみ発行できます。");

        Status = ManufacturingOrderStatus.Issued;
    }
}
```

### 値オブジェクト

- イミュータブル（`init` プロパティのみ）
- 等値比較は値ベース（`record` を使用）
- バリデーションはコンストラクタ内で行う

```csharp
public record LotNumber
{
    public string Value { get; }

    public LotNumber(string value)
    {
        if (string.IsNullOrWhiteSpace(value))
            throw new DomainException("ロット番号は空にできません。");
        if (value.Length > 50)
            throw new DomainException("ロット番号は50文字以内で入力してください。");

        Value = value;
    }
}
```

### ドメイン例外

- ドメインルール違反は `DomainException`（`Synapse.Domain/Exceptions/`）をスローする
- アプリケーション層のビジネスロジック例外は `ApplicationException`（`Synapse.Application/Common/Exceptions/`）を使用する
- `DomainException` は HTTP 422 Unprocessable Entity にマッピングする
- `NotFoundException` は HTTP 404 にマッピングする

```csharp
// Domain
public class DomainException : Exception
{
    public DomainException(string message) : base(message) { }
}

// Application
public class NotFoundException : Exception
{
    public NotFoundException(string name, object key)
        : base($"{name} (ID: {key}) は見つかりませんでした。") { }
}
```

## 6. インフラ層

### EF Core（書き込み）

- **書き込み操作（Command）には EF Core を使用する**
- エンティティの設定は Fluent API で行う（Data Annotations 禁止）
- 設定クラスは `Configurations/` に `IEntityTypeConfiguration<T>` として分離する
- `DbContext` は `AppDbContext` として1つに統一する

```csharp
public class ManufacturingOrderConfiguration
    : IEntityTypeConfiguration<ManufacturingOrder>
{
    public void Configure(EntityTypeBuilder<ManufacturingOrder> builder)
    {
        builder.ToTable("t_manufacturing_orders");
        builder.HasKey(x => x.Id);
        builder.Property(x => x.Id).HasMaxLength(26).IsRequired();
        builder.Property(x => x.ItemCode).HasMaxLength(50).IsRequired();
        builder.Property(x => x.Quantity).HasPrecision(18, 4).IsRequired();
        builder.Property(x => x.Status)
            .HasConversion<string>()
            .HasMaxLength(20)
            .IsRequired();
    }
}
```

### Dapper（読み取り）

- **読み取り操作（Query）には Dapper を使用する**
- 複雑な JOIN・集計クエリが多い読み取りを EF Core の LINQ で表現しようとしない
- SQL は Query ハンドラーまたは専用の ReadModel クラスに記述する
- `IDbConnection` を DI から注入する（接続の生成・破棄はフレームワークに委ねる）

### リポジトリパターン

- インターフェースは `Synapse.Domain/Interfaces/` に定義する
- 実装は `Synapse.Infrastructure/Persistence/Repositories/` に置く
- 汎用リポジトリ（`IRepository<T>`）は使用しない。集約ルートごとに専用インターフェースを定義する

```csharp
// Domain 層で定義
public interface IManufacturingOrderRepository
{
    Task<ManufacturingOrder?> FindByIdAsync(string id, CancellationToken ct);
    Task AddAsync(ManufacturingOrder order, CancellationToken ct);
    Task<bool> ExistsAsync(string id, CancellationToken ct);
}
```

### Unit of Work

- トランザクション管理は `IUnitOfWork` インターフェース経由で行う
- `AppDbContext` が `IUnitOfWork` を実装する
- `TransactionBehavior` がトランザクション開始・コミット・ロールバックを担う

### DBマイグレーション

- EF Core Migrations を使用する
- マイグレーションファイルは必ずレビューしてから Apply する
- 本番環境への Apply は CI/CD パイプラインで `dotnet ef database update` を実行する
- 後退できないマイグレーション（データ削除・カラム削除）は特に慎重に扱う

## 7. API 層

### コントローラー設計

- `[ApiController]` / `[Route("api/[controller]")]` を付与する
- コントローラーは薄いラッパーとして機能する。`_mediator.Send()` を呼ぶだけ
- アクションごとに返却する HTTP ステータスを `[ProducesResponseType]` で明示する

```csharp
[ApiController]
[Route("api/manufacturing-orders")]
[Authorize]
public class ManufacturingOrdersController : ControllerBase
{
    private readonly IMediator _mediator;

    public ManufacturingOrdersController(IMediator mediator)
    {
        _mediator = mediator;
    }

    [HttpGet]
    [ProducesResponseType(typeof(PagedResult<ManufacturingOrderListDto>), StatusCodes.Status200OK)]
    public async Task<IActionResult> GetList(
        [FromQuery] GetManufacturingOrderListQuery query,
        CancellationToken ct)
        => Ok(await _mediator.Send(query, ct));

    [HttpPost]
    [ProducesResponseType(typeof(string), StatusCodes.Status201Created)]
    [ProducesResponseType(StatusCodes.Status422UnprocessableEntity)]
    public async Task<IActionResult> Create(
        [FromBody] CreateManufacturingOrderCommand command,
        CancellationToken ct)
    {
        var id = await _mediator.Send(command, ct);
        return CreatedAtAction(nameof(GetById), new { id }, id);
    }
}
```

### グローバル例外ハンドリング

例外は `ExceptionHandlingMiddleware` で一元変換する。コントローラー内で `try-catch` しない。

| 例外クラス | HTTP ステータス |
|---|---|
| `NotFoundException` | 404 Not Found |
| `DomainException` | 422 Unprocessable Entity |
| `ValidationException`（FluentValidation） | 400 Bad Request |
| `UnauthorizedAccessException` | 401 Unauthorized |
| `ForbiddenAccessException` | 403 Forbidden |
| その他の `Exception` | 500 Internal Server Error |

レスポンス形式（RFC 7807 Problem Details 準拠）：

```json
{
  "type": "https://tools.ietf.org/html/rfc7231#section-6.5.1",
  "title": "Validation Error",
  "status": 400,
  "detail": "リクエストの検証に失敗しました。",
  "errors": {
    "quantity": ["数量は1以上でなければなりません。"]
  }
}
```

### バリデーション

- リクエストのバリデーションは FluentValidation で行う
- バリデータは各 Command / Query と同じフォルダに配置する（`CreateManufacturingOrderCommandValidator.cs`）
- `ValidationBehavior` が自動的に実行し、失敗時は `ValidationException` をスローする

```csharp
public class CreateManufacturingOrderCommandValidator
    : AbstractValidator<CreateManufacturingOrderCommand>
{
    public CreateManufacturingOrderCommandValidator()
    {
        RuleFor(x => x.ItemCode).NotEmpty().MaximumLength(50);
        RuleFor(x => x.Quantity).GreaterThan(0);
        RuleFor(x => x.PlannedStartDate).NotEmpty();
        RuleFor(x => x.PlannedEndDate)
            .GreaterThan(x => x.PlannedStartDate)
            .WithMessage("完了予定日は開始予定日より後でなければなりません。");
    }
}
```

### OpenAPI スペック生成（Microsoft.AspNetCore.OpenApi）

フロントエンドの型・クライアントコードは OpenAPI スペックから自動生成される。
**スペックの品質 = フロントエンドの型の品質** であるため、以下のルールを厳守する。

#### 設定方針

- `Microsoft.AspNetCore.OpenApi`（.NET 10 組み込み）と `Microsoft.Extensions.ApiDescription.Server` を使用する
- ビルド時に `openapi.json` を自動出力し、`development/openapi.json` として Git 管理する
- XML ドキュメント（`<GenerateDocumentationFile>true</GenerateDocumentationFile>`）を有効にし、`/// <summary>` コメントを OpenAPI スペックに含める
- enum は `JsonStringEnumConverter(JsonNamingPolicy.CamelCase)` で camelCase にシリアライズする（Program.cs で一括設定済み）

#### スペック品質ルール

- **全エンドポイントに `[ProducesResponseType]` を必須とする**（省略禁止）
- レスポンス型は具体的な DTO 型を指定する（`typeof(object)` のみの指定は禁止）
- nullable フィールドは必ず `T?` で明示する（nullable 情報がスペックに含まれるため）

#### openapi.json の更新フロー

1. C# の DTO またはコントローラーを変更する
2. `dotnet build` → Swashbuckle が `development/openapi.json` を自動出力する
3. `openapi.json` の変更を Git にコミットする
4. フロントエンド側で `pnpm generate` を実行し、型・クライアントを再生成する

## 8. 認証・認可（JWT / RBAC）

### JWT 認証フロー

1. クライアントがログイン API（`POST /api/auth/login`）を呼び出す
2. 認証成功時にアクセストークン（JWT）とリフレッシュトークンを返す
3. 以降のリクエストは `Authorization: Bearer <token>` ヘッダーで送信する
4. トークン有効期限: アクセストークン 60分 / リフレッシュトークン 7日

### JWT ペイロード

```json
{
  "sub": "user-id",
  "name": "山田 太郎",
  "email": "yamada@example.com",
  "role": "production-manager",
  "iat": 1234567890,
  "exp": 1234567890
}
```

### ロール定義

| ロール | 対象者 |
|---|---|
| `production-manager` | 生産管理担当 |
| `floor-leader` | 製造現場リーダー |
| `floor-worker` | 製造作業員 |
| `quality-control` | 品質管理担当 |
| `warehouse` | 倉庫担当 |
| `shipping` | 出荷担当 |
| `manager` | 管理職 |
| `system-admin` | システム管理者 |

### 認可の実装

- エンドポイントレベルは `[Authorize(Roles = "...")]` で制御する
- ファインドグレイン制御が必要な場合は Policy を定義する（`Authorize("CanApproveConditionalAcceptance")` 等）
- シークレットキーは環境変数 `JWT__Secret` から取得する（コードへのハードコーディング禁止）

## 9. リアルタイム通信（SignalR）

### 対象機能

- 工程進捗のリアルタイム表示（PROC-003）
- 在庫アラート通知（INV-005）

### ハブ構成

| ハブクラス | パス | 用途 |
|---|---|---|
| `ProcessProgressHub` | `/hubs/process-progress` | 工程進捗配信 |
| `NotificationHub` | `/hubs/notifications` | 在庫アラート等の通知 |

### 実装方針

- ハブは `Synapse.API/Hubs/` に配置する
- SignalR への通知は Application 層のハンドラーから直接行わず、`INotificationService` インターフェース経由で行う（Application → Infrastructure の依存を逆転させる）
- 認証済みユーザーのみ接続を許可する（`[Authorize]` を付与）

## 10. バックグラウンドジョブ（Hangfire）

### 対象処理

長時間かかる処理や定期実行はジョブとして実装する。

## 12. 日時の扱い

### 基本方針

| 項目 | 規約 |
|---|---|
| API 送受信フォーマット | **UTC ISO 8601**（例: `2026-03-07T10:00:00Z`） |
| C# 型（タイムスタンプ） | **`DateTimeOffset`**（`DateTime` は使用禁止） |
| C# 型（日付のみ） | **`DateOnly`** |
| DB カラム型（タイムスタンプ） | **`timestamptz`**（`timestamp` は使用禁止） |
| DB カラム型（日付のみ） | **`date`** |

### バックエンド実装ルール

- `DateTime` は使用禁止。理由: タイムゾーン情報が失われ、UTC/ローカル時刻の混入が起きる
- `DateTimeOffset.UtcNow` で現在時刻を取得する（`DateTime.Now` / `DateTime.UtcNow` 禁止）
- EF Core の PostgreSQL マッピング: `timestamptz` ↔ `DateTimeOffset`、`date` ↔ `DateOnly`
- `System.Text.Json` は `DateTimeOffset` を自動的に ISO 8601 でシリアライズする

```csharp
// 禁止
public DateTime CreatedAt { get; private set; }
var now = DateTime.Now;
var now = DateTime.UtcNow;

// 正しい
public DateTimeOffset CreatedAt { get; private set; }
var now = DateTimeOffset.UtcNow;

// 日付のみのフィールド
public DateOnly PlannedDate { get; private set; }
```

### EF Core 設定例

```csharp
builder.Property(x => x.CreatedAt)
    .HasColumnType("timestamptz")
    .IsRequired();

builder.Property(x => x.PlannedDate)
    .HasColumnType("date")
    .IsRequired();
```

## 15. コメント規約（日本語コメント）

### 基本方針

**コードコメントは日本語で書く。**

目的は「業務知識をコードの近くに残す」こと。設計書を都度参照しなくても、コードを読んだだけで「なぜそのような実装にしたか」が分かる状態を維持する。

### 何をコメントするか

#### クラスレベル（`/// <summary>`）

- 業務上の役割（何のデータを保持するか）
- 他のエンティティ・テーブルとの関係（参照される場所）

```csharp
/// <summary>
/// 品目マスタ。製品・原材料・部品の基本情報を保持する。
/// BOM・在庫・製造指示・出荷など、ほぼ全業務テーブルから参照される中核マスタ。
/// </summary>
public class Item : Entity { ... }
```

#### プロパティレベル（`/// <summary>`）

自明でないプロパティには業務用途を書く。特に：
- 他の機能との連携（例: 安全在庫 → `INV-005` の発報条件）
- フラグの具体的な効果（「true にすると〇〇になる」）
- 制約の理由（変更不可・必須等）

```csharp
/// <summary>
/// 安全在庫数。この数量を下回ると在庫アラート（INV-005）が発報される。
/// 0 の場合はアラート対象外。
/// </summary>
public decimal SafetyStockQuantity { get; private set; }

/// <summary>
/// 有効フラグ。false にすると「廃番」扱いとなり、新規の製造指示・発注では選択不可になる。
/// 過去の実績データを残すために物理削除ではなく論理削除を採用している。
/// </summary>
public bool IsActive { get; private set; }
```

#### メソッドレベル（`/// <summary>`）

「何をするか」はメソッド名で表現できるので、コメントには**「なぜそうしているか」（設計判断の理由）**を書く。

```csharp
/// <summary>
/// 品目を廃番にする（論理削除）。
/// 過去の製造実績・在庫履歴に品目名が残るよう、物理削除はしない。
/// 廃番後は製造指示・発注の新規作成で選択できなくなる。
/// </summary>
public void Deactivate() => IsActive = false;
```

#### インラインコメント

ロジックの「なぜ」を補足する。「何をしているか」はコードが語るので書かない。

```csharp
// 品目コードはシステム全体の識別子として使われるため、重複登録を防ぐ。
// DB にもユニーク制約があるが、先に業務例外として返すことでエラーメッセージをわかりやすくする。
var codeExists = await _context.Items.AnyAsync(i => i.Code == request.Code, cancellationToken);
```

```csharp
// 製造指示・発注の品目選択画面では廃番品を表示しないのがデフォルト。
// 廃番品も含めて参照したい場合（マスタ管理画面等）は activeOnly=false で呼ぶ。
if (request.ActiveOnly)
    query = query.Where(i => i.IsActive);
```

#### コントローラー（HTTP ステータスの選択理由）

HTTPステータスコードの選択は設計上の判断を含むため、理由をコメントに残す。

```csharp
// 201 Created + Location ヘッダーで新規リソースの URL を返す（REST の慣習）
return CreatedAtAction(nameof(GetById), new { id }, new { id });
```

### コメントを書かなくていい場合

- コード自体が自明な場合（単純な getter、`return Ok(result)` 等）
- メソッド名・変数名で意図が十分伝わる場合

### コメントを書く言語

| 対象 | 言語 |
|---|---|
| `/// <summary>` XML ドキュメントコメント | 日本語 |
| `//` インラインコメント | 日本語 |
| コード（クラス名・メソッド名・変数名） | 英語（命名規約に従う） |

## 16. 禁止事項まとめ

| 禁止事項 | 理由 |
|---|---|
| Domain 層から他レイヤーへの依存 | クリーンアーキテクチャの根幹ルール |
| Application 層が Infrastructure の具体実装を参照する | 依存性逆転の原則（インターフェース経由にする） |
| コントローラーにビジネスロジックを書く | Application 層（MediatR ハンドラー）に書く |
| EF Core の Data Annotations を使う | Fluent API（`IEntityTypeConfiguration`）に統一する |
| 汎用リポジトリ（`IRepository<T>`）の使用 | 集約ごとに専用インターフェースを定義する |
| 読み取りに EF Core の複雑な LINQ を多用する | Dapper + SQL を使用する |
| 文字列補間でログを書く（`$"..."`） | Serilog の構造化ログを使う |
| JWT シークレットキーをコードにハードコードする | 環境変数から取得する |
| コントローラーで `try-catch` する | `ExceptionHandlingMiddleware` に委ねる |
| 新規ハンドラーにテストを書かない | 必須。省略不可 |
| `[ProducesResponseType]` を省略する | OpenAPI スペックの品質に直結し、フロントエンドの型生成に影響する |
| DTO / コントローラー変更後に `openapi.json` をコミットしない | フロントエンドの生成型との乖離が発生するため |
