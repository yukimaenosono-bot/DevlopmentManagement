# フロントエンド設計方針

> バージョン: 1.0.0
> 作成日: 2026-03-07
> 対象: Synapse フロントエンド開発チーム全員

---

## 1. 目的・スコープ

### 目的

フロントエンド実装が本格化する前に、**全員が従うべきルール**を明文化する。
統一されていない実装は以下の問題を引き起こすため、事前合意が必要：

- コンポーネントの乱立による重複実装・スタイルの不統一
- 命名規約の揺れによる可読性低下
- 状態管理・API通信のパターンが混在することによる保守コストの増大

### このドキュメントで決定すること

- ディレクトリ構成・ファイル命名規約
- 共通UIコンポーネントの一覧と優先度
- 複合コンポーネントの一覧と責務
- デザイントークン（色・スペーシング）の扱い
- 状態管理・API通信・フォームのパターン
- ルーティング・認証・権限制御の方針
- デバイス別UI方針
- テスト方針

### このドキュメントで決定しないこと

- 各コンポーネントの詳細なProps定義（実装時に決定、コード内に型で表現）
- APIエンドポイントの詳細（`detailed-design/05_API仕様書.md` を参照）
- ビジネスロジック・業務ルール（`requirements/` を参照）

## 2. ディレクトリ構成規約

```
development/frontend/src/
├── components/
│   ├── ui/            # 基本UIコンポーネント（shadcn/ui ベース）
│   ├── composite/     # 複数の ui/ を組み合わせた再利用コンポーネント
│   └── layout/        # レイアウト系（Sidebar, Header, PageLayout 等）
├── features/          # 機能ドメイン単位のモジュール
│   └── [domain]/      # 例: manufacturing-order/, inventory/, quality-control/
│       ├── components/ # そのドメイン固有のコンポーネント
│       ├── hooks/      # そのドメイン固有のカスタムhooks
│       └── schemas/    # Zodバリデーションスキーマ（フォームバリデーション用）
├── generated/         # orval が自動生成する API クライアント・型定義（直接編集禁止）
├── hooks/             # グローバルなカスタムhooks（useAuth, useSignalR 等）
├── stores/            # Zustand ストア
├── services/          # axios インスタンス設定のみ（apiClient.ts・queryKeys.ts）
├── types/             # 共通型定義（api-primitives.ts 等）
├── lib/               # ユーティリティ（utils.ts 等）
└── pages/             # ルート対応ページコンポーネント（薄いラッパー）
```

## 6. デザイントークン規約

### カラー

- CSS Variables（OKLch スキーム）を使用する。`index.css` に定義済み
- 色を追加する場合は CSS Variables に追加する（`#RRGGBB` の直接記述禁止）
- ステータスカラーの対応：

| ステータス | Tailwind クラス |
|---|---|
| 正常・完了・合格 | `text-green-600` / `bg-green-100` |
| 警告・保留 | `text-yellow-600` / `bg-yellow-100` |
| エラー・不合格・緊急 | `text-red-600` / `bg-red-100` |
| 情報・進行中 | `text-blue-600` / `bg-blue-100` |
| 無効・キャンセル | `text-gray-400` / `bg-gray-100` |

### スペーシング・タイポグラフィ

- Tailwind のデフォルトスケールのみ使用する
- 任意値（例: `w-[123px]`）は原則禁止。どうしても必要な場合はチームで合意
- フォントサイズは Tailwind のスケール（`text-sm`、`text-base`、`text-lg` 等）を使用

### ブレークポイント

| ブレークポイント | 幅 | 対象端末 |
|---|---|---|
| `sm` | 640px | - |
| `md` | 768px | タブレット基準 |
| `lg` | 1024px | PC基準 |
| `xl` | 1280px | 大型モニター |

- `md` 以上: PCレイアウト
- `md` 未満: タブレット/ハンディレイアウト

| 対象 | 規約 | 例 |
|---|---|---|
| コンポーネントファイル | PascalCase + `.tsx` | `DataTable.tsx`, `BarcodeScanner.tsx` |

| クエリキー定数 | camelCase | `workOrderKeys.all`, `workOrderKeys.detail` |
| CSSクラス | Tailwind クラスのみ。カスタムCSSクラス禁止 | - |

## 8. 状態管理方針（Zustand）

### 原則

| データ種別 | 管理場所 | 理由 |
|---|---|---|
| サーバーデータ（API取得） | **TanStack Query** | キャッシュ・再取得・ローディング状態を自動管理 |
| UI状態・グローバル状態 | **Zustand** | 軽量、シンプル、SSRなし |

### 作成するストア（最小限）

| ストア | ファイル | 役割 | 状態 |
|---|---|---|---|
| `authStore` | `stores/authStore.ts` | ログインユーザー情報・JWTトークン・ロール | 実装済み |
| `uiStore` | `stores/uiStore.ts` | サイドバー開閉状態 | 実装済み |

- ストアは機能追加に応じて慎重に増やす。安易に増やさない
- 両ストアは `zustand/middleware` の `persist` でローカルストレージに永続化済み

## 9. API通信パターン（OpenAPI コード生成 + TanStack Query）

### コード生成フロー（orval）

バックエンドの OpenAPI スペック（`development/openapi.json`）から `orval` で TypeScript 型・React Query hooks を自動生成する。手書きの API 型定義・API クライアント関数は禁止。

```
development/openapi.json（バックエンドビルド時に Microsoft.AspNetCore.OpenApi が自動出力）
    ↓ pnpm generate（orval）
src/generated/
└── [domain名].ts    # 型定義 + axios クライアント関数 + React Query hooks
```

`pnpm generate` を実行するタイミング:
- バックエンドの DTO またはコントローラーを変更したとき
- CI で自動実行し、差分があればビルド失敗とする（型ドリフト防止）

### レイヤー構成

```
コンポーネント / Page
    ↓
features/[domain]/hooks/use[Domain].ts  ← TanStack Query カスタムhook（薄いラッパー）
    ↓
generated/[domain名].ts                 ← orval 生成の API クライアント関数・型定義
    ↓
services/apiClient.ts                   ← axios インスタンス（共通設定）
```

### `services/apiClient.ts` の責務

- ベースURL設定（環境変数から取得）
- リクエストインターセプター: JWT トークンを `Authorization` ヘッダーに自動付与
- レスポンスインターセプター: 401 エラー時の自動ログアウト処理

### クエリキー管理

- 各ドメインのクエリキーは `services/queryKeys.ts` で一元管理
- パターン: `{ all, lists, list, details, detail }` の階層構造

### ルール

- データ取得は `useQuery`、変更系は `useMutation` を使う
- `useMutation` 成功時は `queryClient.invalidateQueries` でキャッシュを更新
- ローディング・エラー状態は TanStack Query の `isPending`・`isError` を使う（独自 state 禁止）
- API 型は `generated/` から import する。`features/` に手書きで型を定義しない

### C# / JSON / TypeScript 型対応早見表

バックエンドの DTO 設計と、orval が生成する TypeScript 型の対応は以下の通り。

| C# 型 | JSON 値の例 | TypeScript 型（生成） |
|---|---|---|
| `string`（ULID） | `"01ARZ3..."` | `string` |
| `string` | `"テキスト"` | `string` |
| `int` / `decimal` | `123` / `123.45` | `number` |
| `bool` | `true` | `boolean` |
| `DateTimeOffset` | `"2026-03-07T10:00:00Z"` | `string` |
| `DateOnly` | `"2026-03-07"` | `string` |
| `enum`（camelCase） | `"issued"` | union type（enum メンバーから自動生成） |
| `T?` | `null` または値 | `T \| null` |
| `IEnumerable<T>` | `[...]` | `T[]` |

## 10. フォーム設計（React Hook Form + Zod）

> **Zod スキーマ自動生成について**: orval が Zod v4 に未対応のため、`src/generated/zod/` への自動生成は無効化済み。
> Zod スキーマは `features/[domain]/schemas/` に**手書きで管理**すること。

### ルール

- 全フォームは `react-hook-form` + `zod` で実装する（`useState` による手動管理禁止）
- Zodスキーマは `features/[domain]/schemas/` に配置し、バックエンドの業務ルールと対応させる
- `FormField` コンポーネントでラベル・入力・エラーメッセージを必ずセットで表示する

### フォームの構成パターン

```tsx
// スキーマ定義
const workOrderSchema = z.object({
  productCode: z.string().min(1, "品目コードは必須です"),
  quantity: z.number().int().positive("数量は1以上の整数を入力してください"),
  plannedDate: z.date(),
})

// フォームコンポーネント内
const form = useForm({ resolver: zodResolver(workOrderSchema) })
```

## 12. 認証・権限制御（RBAC）

### 認証フロー

1. ログイン成功 → JWTトークン取得 → `authStore` に保存
2. axios インターセプターが全リクエストにトークンを付与
3. ページ遷移時に `authStore` のトークン有効期限を確認

### ロール定義

| ロール | 対象者 | 主な端末 |
|---|---|---|
| `production-manager` | 生産管理担当 | PC |
| `floor-leader` | 製造現場リーダー | タブレット |
| `floor-worker` | 製造作業員 | タブレット・ハンディ |
| `quality-control` | 品質管理担当 | PC・タブレット |
| `warehouse` | 倉庫担当 | ハンディ |
| `shipping` | 出荷担当 | PC・ハンディ |
| `manager` | 管理職 | PC |
| `system-admin` | システム管理者 | PC |

### 権限制御の実装

- `usePermission(role)` hook で表示制御を行う
- ルートレベル: 未認証ユーザーは `/login` にリダイレクト
- 画面レベル: 権限なしは 403 画面を表示
- ボタン・フィールドレベル: `usePermission` で `disabled` または非表示

## 13. デバイス別UI方針

| 端末 | レイアウト | UX指針 |
|---|---|---|
| PC | サイドバー常時表示、通常フォントサイズ | 情報密度重視、キーボード操作対応 |
| タブレット | ヘッダーのみ、大フォント・大タップ領域 | タップ領域 min 48px、バーコードスキャン対応 |
| ハンディターミナル | 縦持ちシングルカラム | 最小限の操作のみ、大きなボタン |

### タブレット・ハンディ固有の実装指針

- タップ対象要素は最低 `min-h-12`（48px）を確保する
- フォームの数値入力には `inputMode="numeric"` を付与してテンキーを表示する
- バーコードスキャン機能は `BarcodeScanner` コンポーネントを必ず共通利用する（個別実装禁止）
- SignalR リアルタイム更新が必要な画面では `useSignalR` hook を使う

## 15. 日時の扱い

### 基本方針

| 項目 | 規約 |
|---|---|
| API から受け取る型 | `string`（ISO 8601 UTC 文字列） |
| API に送る型 | `string`（ISO 8601 UTC 文字列） |
| 画面表示 | ユーティリティ関数経由でローカル時刻に変換して表示 |
| `Date` オブジェクト | 表示変換の内部のみで使用。コンポーネントには渡さない |

### TypeScript 型定義ルール

API レスポンスの型は `generated/` から import する（`features/` に手書き定義しない）。
`types/api-primitives.ts` の型エイリアスは、フォームスキーマやユーティリティ関数の引数型として使用する。

```typescript
// NG: 手書きで API レスポンス型を定義する
import type { IsoDateTimeString } from '@/types/api-primitives'
type ManufacturingOrderListItem = { id: string; plannedStartDate: IsoDateTimeString }

// OK: generated/ から import する
import type { ManufacturingOrderListDto } from '@/generated/manufacturing-orders'

// OK: ユーティリティ関数の引数には型エイリアスを使う（意図を明示）
import type { IsoDateTimeString } from '@/types/api-primitives'
export function formatDateTime(utcString: IsoDateTimeString): string { ... }
```

orval が生成する型では `DateTimeOffset` / `DateOnly` はどちらも `string` になる。`Date` 型の使用は引き続き禁止。

### 表示用ユーティリティ（`lib/utils.ts`）

日時の表示には必ず以下の関数を経由する。`new Date().toLocaleDateString()` の直接呼び出し禁止。

```typescript
// UTC 文字列 → ローカル日時表示（"2026/03/07 10:00"）
export function formatDateTime(utcString: string): string {
  return new Intl.DateTimeFormat("ja-JP", {
    year: "numeric", month: "2-digit", day: "2-digit",
    hour: "2-digit", minute: "2-digit",
    timeZone: "Asia/Tokyo",
  }).format(new Date(utcString))
}

// UTC 文字列 → ローカル日付表示（"2026/03/07"）
export function formatDate(dateString: string): string {
  return new Intl.DateTimeFormat("ja-JP", {
    year: "numeric", month: "2-digit", day: "2-digit",
    timeZone: "Asia/Tokyo",
  }).format(new Date(dateString))
}
```

### フォーム送信ルール

- `DatePicker` の値を API に送るときは UTC ISO 8601 文字列に変換してから送信する
- Zod スキーマの日時フィールドは `z.string().datetime()` で定義する

```typescript
const workOrderSchema = z.object({
  plannedStartDate: z.string().datetime(),   // UTC ISO 8601 を強制
  plannedDate: z.string().date(),            // "YYYY-MM-DD" 形式
})
```

## 16. 禁止事項まとめ

| 禁止事項 | 理由 |
|---|---|
| 色を `#RRGGBB` で直接記述する | デザイントークン（CSS Variables）で統一するため |
| Tailwind の任意値（`w-[123px]`）を多用する | スケール統一のため |
