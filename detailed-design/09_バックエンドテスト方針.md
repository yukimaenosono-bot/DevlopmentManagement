# 09. バックエンドテスト方針

## 概要

このドキュメントでは、Synapse バックエンドのテスト戦略・書き方・命名規則を定める。

---

## テスト戦略

### 何をテストするか

| 対象 | テスト種別 | 理由 |
|------|----------|------|
| CommandHandler / QueryHandler | **ユニットテスト** | ビジネスロジックの中心。必須 |
| Domain エンティティ | **ユニットテスト** | 値の検証・状態遷移を確認 |
| API エンドポイント（Controller） | **統合テスト**（将来） | ルーティング・HTTPステータス確認 |

### 何をテストしないか

| 対象 | 理由 |
|------|------|
| Infrastructure 実装（EF Core, IdentityService） | DBや外部サービスに依存。モックで間接確認 |
| Controller 本体 | MediatR 経由なので Handler のテストで代替 |
| DI の設定 | フレームワーク側の責任 |

---

## テストプロジェクト構成

```
tests/Synapse.Tests/
├── Application/
│   ├── Auth/
│   │   └── Commands/
│   │       └── LoginCommandHandlerTests.cs
│   ├── Items/           ← 品目マスタ追加時に作成
│   │   └── Commands/
│   │       └── CreateItemCommandHandlerTests.cs
│   └── ...
└── Domain/              ← Domainエンティティのテスト（将来）
```

テストの配置は **本体コードと同じフォルダ構成** にする。

## 命名規則

### クラス名

```
{対象クラス名}Tests
例: LoginCommandHandlerTests
```

### メソッド名

```
{メソッド名}_{条件}_{期待結果}
例: Handle_ValidCredentials_ReturnsToken
    Handle_InvalidPassword_ThrowsUnauthorizedAccessException
```

## ユニットテストの書き方

### 基本構造（AAA パターン）

```csharp
[Fact]
public async Task Handle_ValidCredentials_ReturnsToken()
{
    // Arrange（準備）
    var mockIdentityService = new Mock<IIdentityService>();
    mockIdentityService
        .Setup(s => s.LoginAsync("admin", "Admin1234"))
        .ReturnsAsync((true, "jwt-token-here"));

    var handler = new LoginCommandHandler(mockIdentityService.Object);
    var command = new LoginCommand("admin", "Admin1234");

    // Act（実行）
    var result = await handler.Handle(command, CancellationToken.None);

    // Assert（検証）
    result.Token.Should().Be("jwt-token-here");
}
```

### 例外ケースの検証

```csharp
[Fact]
public async Task Handle_InvalidPassword_ThrowsUnauthorizedAccessException()
{
    // Arrange
    var mockIdentityService = new Mock<IIdentityService>();
    mockIdentityService
        .Setup(s => s.LoginAsync(It.IsAny<string>(), It.IsAny<string>()))
        .ReturnsAsync((false, string.Empty));

    var handler = new LoginCommandHandler(mockIdentityService.Object);

    // Act & Assert
    await handler
        .Invoking(h => h.Handle(new LoginCommand("admin", "wrong"), CancellationToken.None))
        .Should().ThrowAsync<UnauthorizedAccessException>();
}
```

### IApplicationDbContext のモック例（Command が DB を使う場合）

```csharp
// DbSet をモックするヘルパー
private static Mock<DbSet<T>> CreateMockDbSet<T>(IQueryable<T> data) where T : class
{
    var mock = new Mock<DbSet<T>>();
    mock.As<IQueryable<T>>().Setup(m => m.Provider).Returns(data.Provider);
    mock.As<IQueryable<T>>().Setup(m => m.Expression).Returns(data.Expression);
    mock.As<IQueryable<T>>().Setup(m => m.ElementType).Returns(data.ElementType);
    mock.As<IQueryable<T>>().Setup(m => m.GetEnumerator()).Returns(data.GetEnumerator());
    return mock;
}

// 使い方
var items = new List<Item> { new Item { ... } }.AsQueryable();
var mockDbSet = CreateMockDbSet(items);

var mockContext = new Mock<IApplicationDbContext>();
mockContext.Setup(c => c.Items).Returns(mockDbSet.Object);
```

## テスト実行

```bash
# development/backend/ ディレクトリで実行
dotnet test

# 特定のテストクラスだけ実行
dotnet test --filter "ClassName=LoginCommandHandlerTests"

# カバレッジ付き実行
dotnet test --collect:"XPlat Code Coverage"
```

---

## ルール

1. **新しい CommandHandler / QueryHandler を追加したら必ずテストも追加する**
2. テストは `[Fact]`（単一）または `[Theory] + [InlineData]`（複数パターン）を使う
3. モックは `Mock<T>` を使い、`It.IsAny<T>()` で汎用化する
4. アサーションは `FluentAssertions`（`.Should().Be()` 等）で書く
5. テストが通らない状態でコミットしない（`dotnet test` を先に実行する）
